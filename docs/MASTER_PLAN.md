## 1. 项目目标
构建一个可独立部署、低耦合集成到复杂企业系统中的可观测性工具包。

核心结果：
- 让日志、指标、链路在日常排障中真正可用。
- 统一日志字段规范和查询方法。
- 提供可复用的仪表盘与告警规则。
- 降低生产故障的平均恢复时间（MTTR）。

## 2. 范围定义
范围内（MVP 到 V1）：
- 基于本地进程脚本（非 Docker 主路径）的可观测栈：Prometheus + Grafana + Loki + Promtail + Tempo + OTel Collector。
- 统一日志字段规范与迁移说明。
- LogQL 查询手册（高频排障模板）。
- API 健康与错误定位相关的 Grafana 仪表盘。
- 基础 SLO/SLA 信号告警规则。

暂不纳入：
- 完整生产级高可用部署（Helm、多可用区、长期存储）。
- 一步替换现有第三方日志平台。
- 复杂 AIOps 自动化闭环。

## 3. 工作原则
- 独立仓库、独立版本节奏。
- 先“可用”，再“完美”。
- 先统一字段，再做复杂可视化。
- 每个里程碑都必须包含：可运行产物、文档、验证清单。

## 4. 里程碑
## M0 - 仓库初始化（1-2 天）
交付物：
- 项目骨架：`compose/`、`configs/`、`dashboards/`、`docs/`、`scripts/`。
- 启动说明 README。
- 环境变量模板文件。
- 本地启动/验证脚本（PowerShell）。

验收标准：
- 执行本地启动脚本可启动核心服务（非 Docker）。
- Grafana 与 Prometheus 页面可访问。

## M1 - 日志可用性基线（1 周）
交付物：
- 日志字段规范：`service`、`module`、`env`、`traceId`、`spanId`、`errorCode`、`userId`、`path`、`latencyMs`。
- Promtail 解析流水线示例。
- LogQL v1 查询手册（不少于 10 条常用查询）。
- “Top 5 排障剧本”文档。

验收标准：
- 能通过 `traceId` 定位单次请求。
- 能按 `errorCode` 与路由聚合错误。
- 能过滤噪声日志且不丢失关键信号。

## M2 - 指标与告警基线（1 周）
交付物：
- Prometheus 抓取配置模板。
- 两个核心仪表盘：
  - API 健康看板（QPS、p95/p99 延迟、5xx 比例）
  - 错误定位看板（错误趋势、Top 接口、Top 服务）
- 初版告警规则：
  - 5xx 比例异常
  - p95 延迟超阈值
  - 实例不可用

验收标准：
- 告警可在测试环境触发并恢复。
- 值班人员可用仪表盘完成日常故障定位。

## M3 - 链路关联（1 周）
交付物：
- OTel Collector 基线配置。
- Tempo 接入与 Grafana 链路面板。
- 日志与链路关联说明文档。

验收标准：
- 可在采样请求中实现日志与链路互相跳转定位。

## M4 - 企业集成试点（1-2 周）
交付物：
- 接入公司环境中的一个真实服务。
- 差距报告：字段缺口、查询痛点、成本与风险。
- V1 加固待办清单。

验收标准：
- 真实故障场景的定位速度优于现有基线。

## 6. 缺陷与变更流程
- 严重级别：
  - S0：数据丢失或故障期间无可观测能力
  - S1：关键仪表盘或查询不可用
  - S2：局部功能异常
  - S3：文档或易用性问题
- 响应时效：
  - S0：立即热修复
  - S1：当天修复
  - S2：下个迭代处理
  - S3：进入待办池
- 每次修复必须包含：
  - 根因分析
  - 配置/代码差异
  - 验证步骤
  - 文档更新
