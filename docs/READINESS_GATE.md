# Ops Observability Kit - 实施前门禁（Readiness Gate）

最后更新：2026-03-02

## 目标
在开始任何代码实现前，确保范围、依赖、风险、验收和协作机制已闭环，避免“边做边返工”。

## 当前门禁模式
- 模式：单人开发轻门禁（生效日期：2026-03-02）
- 适用范围：单人独立开发阶段
- 说明：跨团队审批与“全部依赖 READY”不再作为编码启动前置，改为“当前任务可执行 + 风险可控”。

## 门禁定义
- G0 范围冻结：`MASTER_PLAN.md`、`TASK_BREAKDOWN.md` 本周版本冻结。
- G1 依赖可用：当前会话目标所需依赖可执行；非当前目标依赖可处于 `PENDING/BLOCKED`。
- G2 可验证性：每个 P0 任务都有量化验收标准与验证命令。
- G3 风险有预案：S0 必须可回滚；S1/S2 允许在有绕行方案或风险接受时并行推进。
- G4 协作准备：负责人与响应 SLA 明确。
- G5 启动批准：单人开发者自批准并记录依据。

## Go / No-Go 判定
Go（可开始实现）必须同时满足：
1. G0、G2、G5 通过。
2. 不存在未受控的 S0 风险。
3. 当前任务的阻碍/依赖具备可执行绕行方案或已记录 `ACCEPTED_RISK`。
4. `SESSION_PLAYBOOK.md` 已确认可按步骤执行。

No-Go（不得开始实现）任一命中即成立：
1. 存在未受控 S0 风险。
2. 当前会话目标无法执行且无绕行方案。
3. 验收标准无法量化且无法补充验证动作。
